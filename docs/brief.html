Veritas - the AI never gets the final word

The trust layer for AI investigations: deterministic code, not the model, decides what is "confirmed," and every finding traces by foreign key to the exact forensic tool that proved it.

The problem

High-stakes work adopts AI fastest - incident response, fraud, compliance - and that is exactly where you cannot ship a conclusion you cannot audit. Courts, insurers, and auditors do not trust the model. Veritas makes the trust provable.

Why it is original

"Code overrules the AI, and every claim traces to proof" is not a slogan here - it is a SQL schema you can query. The override is a real column, the proof a real foreign key, the corroboration a real ON CONFLICT merge.

What it does

  • Case dashboard - verdict, SHA-256 evidence-integrity badge, five-way disposition scoreboard.
  • AI-overruled view - where the model wanted CONFIRMED MALICIOUS but the code said SUSPICIOUS, with the exact gate that withheld promotion.
  • Proof chain - one click: claim to validated fact to source forensic tool, with the raw tool output.
  • Cross-case IOC pivot - type a hash, IP, or PID; one indexed Aurora query returns every case it appears in.
  • Queue a new investigation - Postgres-as-queue; a worker claims jobs with SELECT ... FOR UPDATE SKIP LOCKED and streams 16-step progress.

How Amazon Aurora PostgreSQL is used (the data model is the product)

  • Foreign keys enforce the chain of custody - a finding cannot cite proof that does not exist.
  • UNIQUE(case_id, fact_signature) turns the engine's SHA-1 dedup into an idempotent ON CONFLICT UPSERT - cross-tool corroboration in one statement.
  • ~19 real indexes (btree, GIN on JSONB, pg_trgm) mirror the engine's pivot indexes; a recursive CTE walks the process tree; a materialized view powers cross-case pivots.
  • RLS by org_id is scaffolded for multi-tenant SaaS; Serverless v2 scales to zero between investigations. The data is real - the ingest adapter asserts DB counts match the engine exactly.
